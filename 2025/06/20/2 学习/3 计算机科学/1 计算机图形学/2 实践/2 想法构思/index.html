初步构想

写一个三维的空间，用户可以自由操控摄像机移动，观察不同的物体。

需求分析

功能性

  • 场景渲染: 系统必须能够在窗口中渲染一个由多个3D物体构成的场景。
  • 相机系统: 系统需提供一个第一人称相机，用户可以通过键盘（W/A/S/D）和鼠标移动来自由浏览场景。
  • 数据驱动: 场景的布局（物体的类型、位置、旋转、缩放等）必须能够从外部数据文件（如JSON）动态加载，而非在代码中硬编码。
  • 材质多样性: 系统应支持为不同物体应用不同的着色器（Shader），以实现多样的视觉效果。
  • 资源复用: 对于几何形状相同的多个物体（如多个立方体），系统应只在内存和显存中保留一份网格（Mesh）数据。

拓展性

  • 模块化: 各个子系统（渲染、资源管理、场景）应高度解耦，降低依赖。
  • 可扩展性: 添加新的物体类型、几何形状或着色器效果应足够简单，且不影响现有系统。
  • 性能: 渲染循环应足够高效，通过减少不必要的API调用（如渲染批处理）来保证流畅的帧率。
  • 资源安全: 系统必须妥善管理内存与GPU资源，避免内存泄漏或资源悬挂。
  • 可维护性: 代码结构应清晰、遵循设计模式，并有明确的职责划分，便于理解和修改。

系统设计

为满足上述需求，系统采用分层架构，遵循“关注点分离”的核心设计原则。

架构分层

  1. 应用层 (Application Layer): 最高层，负责整体流程控制。
    • 组件: Application
    • 职责: 初始化系统、管理窗口、驱动主循环、分发用户输入。作为所有下层模块的协调者。
  2. 场景层 (Scene Layer): 描述“世界中有什么”。
    • 组件: World, Object, Camera, ObjectFactory, SceneSerializer
    • 职责: 管理场景中所有对象的集合与逻辑关系。定义对象的变换，并组织对象以便高效渲染。实现从数据到场景的转换。
  3. 资源层 (Resource Layer): 负责资源的生命周期管理与共享。
    • 组件: MeshManager, ShaderManager
    • 职责: 抽象化资源的加载、缓存和分发。确保资源高效复用。
  4. 渲染抽象层 (Rendering Abstraction Layer): 封装底层图形API（OpenGL）的具体操作。
    • 组件: Mesh, Shader
    • 职责: 将GPU资源（顶点缓冲、着色器程序）封装为面向对象的C++类，向上层提供简洁的调用接口（如Bind(), setMat4()）。

核心数据流

  1. 启动: Application初始化，并指令Resource Layer（MeshManager, ShaderManager）预加载通用资源。
  2. 场景加载: Application调用SceneSerializer。SceneSerializer解析scene.json文件。
  3. 对象实例化: SceneSerializer根据JSON数据，向ObjectFactory请求创建对象。Factory则向Resource Layer请求共享的Mesh和Shader资源，并用这些资源实例化具体的Object。
  4. 世界构建: Object被移入World中。World根据其Shader指针，将对象分拣到不同的渲染批次中。
  5. 渲染: Application在主循环中，调用World的渲染方法。World高效地遍历其渲染批次，执行渲染指令。

详细设计

类与职责

  • Application:
    • 职责: 统筹全局。拥有World, ShaderManager, MeshManager的实例。处理GLFW回调，更新相机，调用World::Render()。
  • ShaderManager / MeshManager:
    • 设计模式: 管理器模式 (Manager Pattern) / 缓存模式 (Cache Pattern)。
    • 职责:
      • ShaderManager: 内部使用std::map<string, unique_ptr<Shader>>缓存着色器，保证每个Shader只编译一次，并管理其生命周期。
      • MeshManager: 内部使用std::map<string, shared_ptr<Mesh>>缓存网格。通过shared_ptr实现飞量模式 (Flyweight Pattern)，允许多个Object共享同一个Mesh实例。
  • Object (基类) / Cuboid (派生类):
    • 职责:
      • Object: 场景中的基本单元。不拥有GPU资源，而是持有来自管理器的std::shared_ptr<Mesh>和Shader*。存储自身的变换矩阵（model）。
      • Cuboid: Object的具体实现，代表一种几何类型。
  • ObjectFactory:
    • 设计模式: 工厂模式 (Factory Pattern)。
    • 职责: 解耦对象的创建过程。根据一个type字符串，向MeshManager请求对应的网格，并创建相应的Object派生类实例。
  • World:
    • 职责: 场景对象的容器与渲染调度器。
    • std::vector<unique_ptr<Object>> m_Objects: 主列表，通过unique_ptr明确对Object的所有权。
    • std::map<Shader*, vector<Object*>> m_RenderBatches: 核心性能优化点。实现渲染批处理，将使用相同Shader的对象归为一类，渲染时批量处理，避免频繁切换glUseProgram。
  • SceneSerializer:
    • 职责: 数据驱动的核心。负责I/O，解析JSON，并作为客户端调用ObjectFactory和World的接口来构建场景。
  • Mesh:
    • 职责: RAII（资源获取即初始化）封装。在构造函数中创建OpenGL对象（VAO, VBO, EBO），在析构函数中释放它们。提供Bind()/Unbind()接口。
  • Camera:
    • 职责: 封装相机的所有逻辑，包括位置、方向向量、欧拉角，以及GetViewMatrix()的计算和键盘/鼠标输入的处理。

所有权与资源管理

  • 明确所有权: 大量使用智能指针来自动化资源管理，杜绝内存泄漏。
    • unique_ptr: World对Object拥有独占所有权。ShaderManager对Shader拥有独占所有权。
    • shared_ptr: MeshManager创建Mesh后，将其以shared_ptr分发给多个Object共享。
