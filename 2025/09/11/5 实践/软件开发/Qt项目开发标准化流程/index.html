FAQ

在 Qt 的 MinGW、Build Tools 里的、独立安装的 MinGW 的区别是什么？

Build Tools 中的 MinGW 是一个“特供版”，它经过 Qt 官方的测试和配置，保证与该版本的 Qt 无缝协作；而自己独立安装的 MinGW 是一个“通用版”，版本可能更新，但需要手动配置才能与 Qt 正常工作。

Qt Design Studio 和 Qt Creator 的区别

特性 Qt Design Studio Qt Creator 内的 UI 设计器
目标用户 UI/UX 设计师 程序员
核心焦点 视觉表现、动画、交互、原型制作 功能实现、快速布局、信号与槽连接
动画能力 非常强大 (时间轴、关键帧) 非常有限 (主要通过属性绑定)
3D 支持 原生集成，可进行 3D 场景编辑 基本不支持，需要手写 QML 代码
设计导入 支持 (Figma, Photoshop 等) 不支持
最终产出 高质量的 QML 代码 (定义了界面的外观和感觉) QML 或 .ui 文件 (定义了界面的结构和基本属性)

CDB Debugger Support 和 Debugging Tools for Windows 是做什么的？

CDB Debugger Support 是一个让 Qt Creator 能够在 Windows 系统上、在使用微软 MSVC 编译器时，具备代码调试功能的核心组件。

没有它，在 Windows 上用 MSVC 编译代码时，Qt Creator 里的“调试”按钮就等于废了。

  • CDB (MSVC)：CDB 是 Windows 操作系统的一部分，由微软通过 “Debugging Tools for Windows” 这个独立的包来分发。Qt 本身不包含它，所以 Qt 安装器需要一个额外的选项来提醒你，并帮你从微软那里下载安装。

  • GDB (GCC)：GDB 和 GCC 编译器本身就是一家人，它们通常被打包在一起。当你下载 MinGW 工具链时，里面不仅包含了编译器 g++.exe，通常也已经包含了调试器 gdb.exe。

什么是 Ninja？

Ninja 是一个专注于极致速度的小型、现代化的构建系统。它本身不负责规划项目如何编译的复杂逻辑，而是像一个高效的“施工队”，专门负责执行由 CMake 这样的“项目经理”所生成的简单、明确的编译指令。其核心优势在于极低的开销和出色的并行处理能力，能够充分利用多核CPU，从而显著缩短大型项目的编译等待时间，尤其是在只修改了少量文件的增量构建中表现卓越。因此，Ninja 通常与 CMake 搭档使用，作为其后端，共同构成了一套高效、快速的现代C++开发构建方案。

为什么MinGW需要单独安装，但是MSVC不用？

特性 MinGW MSVC
提供方 开源社区 (GNU) 微软公司
获取方式 可由 Qt 安装程序捆绑安装 必须由 Visual Studio 安装程序独立安装
与 Qt 的关系 作为便利选项被集成 作为一个系统前提被检测和支持
本质 一个独立的、可自由分发的工具 一个与操作系统和 IDE 深度绑定的平台工具

Qt 公司无权把微软的 MSVC 集成，但是可以集成开源的 MinGW

配置 CMakeLists.txt

首先是找到 python 环境

# --- python interpreter ---
set(PYTHON_ENV_SOURCE_DIR ${CMAKE_CURRENT_SOURCE_DIR}/../backend/env/python)
if(NOT EXISTS ${PYTHON_ENV_SOURCE_DIR})
    message(FATAL_ERROR "Python environment directory not found at ${PYTHON_ENV_SOURCE_DIR}")
endif()

install(
        DIRECTORY ${PYTHON_ENV_SOURCE_DIR}/
        DESTINATION .          # install on root
        COMPONENT Runtime
        # exclude cache
        PATTERN "*.pyc" EXCLUDE
        PATTERN "__pycache__" EXCLUDE
)
# -----------------

此时 CMAKE_CURRENT_SOURCE_DIR 就是嵌入的 python 环境路径

此时再设置 python 解释器的路径即可让 pybind 找到解释器

# C++ invoke python
set(Python_EXECUTABLE ${PYTHON_ENV_SOURCE_DIR}/python.exe)

然后配置自动拉取，构建时自动检查 pybind，如果没有则下载。

include(FetchContent)
FetchContent_Declare(
        pybind11
        GIT_REPOSITORY https://github.com/pybind/pybind11.git
        GIT_TAG        v3.0.1
)
FetchContent_MakeAvailable(pybind11)

target_link_libraries(cpp_py_test PRIVATE pybind11::embed)

然后添加链接库 pybind

前面的 Qt 是默认的，只需配置最后一个

target_link_libraries(pce_ui
        Qt::Core
        Qt::Gui
        Qt::Widgets
        pybind11::embed
)

添加外部库

对于发布在 github 上的库，最佳实践如下：

在 CMake 高于 3.11 的版本中，拥有自动从 github 拉取并且配置到本地的能力。使用为在 CMakeList 文件中添加下面这样的代码：

include(FetchContent)    # 导入 FetchContent 功能，只需要一次

FetchContent_Declare(
    <local_name>         # 给这个库起一个本地的名字，比如 spdlog, glfw
    GIT_REPOSITORY <URL> # 仓库的 Git URL
    GIT_TAG        <tag> # 版本号、分支或 commit hash
)

FetchContent_MakeAvailable(<local_name>) # 构建时自动检查库是否存在，如果不存在就拉取

例如使用 pybind 在 C++ 中调用 python 的代码

include(FetchContent)
FetchContent_Declare(
        pybind11
        GIT_REPOSITORY https://github.com/pybind/pybind11.git
        GIT_TAG        v3.0.1
)
FetchContent_MakeAvailable(pybind11)
